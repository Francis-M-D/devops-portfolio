1. MindTrack CI/CD Pipeline

AWS CodePipeline + Kubernetes Architecture

📌 Project Overview

Designed and implemented a complete CI/CD pipeline using AWS managed services and Kubernetes orchestration. This project demonstrates enterprise-grade automation with zero-manual deployment intervention.

AWS CodePipeline Pipeline orchestration & automation

🚀 Pipeline Architecture

  1. Source Stage - Developer pushes code to GitHub repository
  2. Detection - CodePipeline detects changes via webhook (instant trigger)
  3. Build Stage - CodeBuild compiles code, runs unit tests, and generates artifacts
  4. Containerization - Docker image is built with production optimizations
  5. Registry - Docker image pushed to Amazon ECR with versioning
  6. Deployment - Kubernetes deployment updated automatically on EKS cluster
  7. Verification - Health checks and smoke tests validate deployment

💡 Key Implementations

Webhook Trigger

Configured GitHub webhook to automatically trigger CodePipeline on code commits, eliminating manual deployment triggers.

# CodePipeline webhook listens for GitHub push events # Automatically starts pipeline when changes detected # Reduces deployment initiation time to ~5 seconds

Multi-stage Build

CodeBuild performs:

  • Code compilation and dependency resolution
  • Unit and integration testing
  • Code quality analysis
  • Security scanning

Docker Optimization

Implemented multi-stage Docker build for minimal image size:

FROM node:18-alpine as builder # Build stage RUN npm install && npm build FROM node:18-alpine # Runtime stage (only includes necessary files) COPY --from=builder /app/dist /app/dist

EKS Deployment

Automated Kubernetes deployment with:

  • Rolling updates (zero-downtime deployments)
  • Health checks and readiness probes
  • Resource limits and requests
  • Auto-scaling policies

🎯 Challenges & Solutions

⚠️ Challenge: Image Size Bloat
✅ Solution Used multi-stage Docker builds, reducing image size from 850MB to 150MB, resulting in 40% faster push to ECR and deployment time reduction.
⚠️ Challenge: Deployment Consistency
✅ Solution Implemented Docker image versioning with build timestamps and git commit SHAs, ensuring traceability and easy rollback capability.
⚠️ Challenge: Zero-downtime Updates
✅ Solution Configured Kubernetes rolling updates with proper health checks, allowing new pods to become ready before terminating old ones.

📚 Key Learnings

AWS Managed Services Philosophy

  • CodePipeline abstracts away infrastructure management
  • Managed services reduce operational overhead
  • Built-in monitoring and logging integrate seamlessly

Container Orchestration at Scale

  • EKS handles complex networking and service discovery automatically
  • Horizontal pod autoscaling provides cost optimization
  • Node auto-scaling ensures resource efficiency

CI/CD Best Practices

  • Webhook-driven automation eliminates manual steps
  • Automated testing prevents bad code from reaching production
  • Artifact versioning enables fast rollback

Jenkins + Docker + Terraform + Ansible

🚀 Complete Automation Flow

Stage 1: Code Trigger

Developer pushes to either dev or master branch

  • GitHub webhook automatically triggers Jenkins job
  • Webhook provides branch information to Jenkins
  • Different deployment paths for dev vs production

Stage 2: Jenkins Pipeline

pipeline { stages { stage('Checkout') { steps { git 'https://github.com/.../repo.git' } } stage('Build') { steps { sh 'npm install && npm run build' } } stage('Docker Build') { steps { sh 'docker build -t app:latest .' } } stage('Push') { steps { sh 'docker push dockerhub/app' } } stage('Deploy') { steps { ansiblePlaybook 'deploy.yml' } } } }

Stage 3: Docker Containerization

Multi-stage Docker build optimized for React:

  • Build stage: Compiles React app with Webpack
  • Runtime stage: Lightweight Nginx image (only ~50MB)
  • Environment variables configurable at runtime

Stage 4: Terraform Provisioning

Infrastructure setup includes:

  • EC2 instances for dev and production
  • VPC with public/private subnets
  • Security groups with SSH and HTTP rules
  • Auto-scaling groups for production
  • Load balancer for traffic distribution

Stage 5: Ansible Deployment

Dynamic configuration and deployment:

--- - hosts: all tasks: - name: Install Docker shell: curl -fsSL https://get.docker.com | sh - name: Pull Docker image docker_image: name: "{{ docker_image }}" source: pull - name: Deploy container docker_container: name: react-app image: "{{ docker_image }}" ports: - "80:3000" state: started - name: Update security group ec2_group: name: app-sg rules: "{{ sg_rules }}"

Stage 6: Security & Access

  • HTTP access (port 80) - Public, allows app traffic
  • SSH access (port 22) - Restricted to specific IPs only
  • Principle of least privilege enforced
  • Automatic security group updates via Ansible

💡 Key Automation Features

Environment-based Deployments

  • Dev branch → Development environment (immediate deployment)
  • Master branch → Production environment (requires approval)
  • Separate Docker Hub repositories for each environment
  • Environment variables injected at container startup

Bash Script Automation (60% manual effort reduction)

Custom scripts handle:

  • Build optimization
  • Image tagging and versioning
  • Push to Docker Hub
  • Health checks and rollback logic

Dynamic Ansible Inventory

Automatically discovers EC2 instances:

plugin: aws_ec2 regions: - us-east-1 filters: tag:Environment: production instance-state-name: running

Infrastructure Scaling

  • Terraform auto-scaling groups for production
  • Load balancer distributes traffic
  • Automatic instance creation/termination based on demand
  • Cost optimization through right-sizing

🎯 Challenges & Solutions

⚠️ Challenge: Branch-based Deployment Logic
✅ Solution Implemented Jenkins conditional steps that read git branch information and route deployments to appropriate environment.
⚠️ Challenge: Dynamic IP Address Changes
✅ Solution Created Ansible playbooks that dynamically update security group rules based on current environment variables, allowing SSH access from changing IPs.
⚠️ Challenge: React App Port Mapping
✅ Solution Configured Nginx reverse proxy in Docker container to expose production build on port 80, while keeping Node.js on internal port 3000.
⚠️ Challenge: Zero-downtime Updates
✅ Solution Implemented blue-green deployment strategy where new version is tested on separate EC2 instance before load balancer switches traffic.
